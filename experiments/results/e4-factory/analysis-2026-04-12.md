# E4 Comparative Analysis

Date: `2026-04-12`

## Question

For the bounded CRM factory slice, is `prompt-language` better than direct Codex, and if not, what
is it better or worse at?

## Dataset

Runs considered:

- `A02-crm-http-headless`
- `20260412-0916-a03-core-proof-prebootstrapped`
- `20260412-1005-a04-core-proof-sequential`
- `20260412-1047-a05-core-proof-sequential-patched`

Primary references:

- [research-method.md](./research-method.md)
- [comparison.md](./comparison.md)

## Main Findings

### 1. Product capability is at least at parity on the bounded CRM core slice

In `A03`, both `prompt-language` and direct Codex completed the same bounded CRM core slice and
passed `lint`, `typecheck`, and `test`.

That means the current evidence does **not** support the claim that prompt-language blocks or
degrades core product completion for this scope.

### 2. Direct Codex was better on operational simplicity in the clean A/B run

In `A03`, the direct Codex baseline ran with less launcher/path friction and cleaner top-level
evidence files. Prompt-language needed Windows-specific launcher fixes, state-root fixes, and had
misleading run-node timeout behavior before the end-state evidence was stable.

So for a single bounded slice with the same model family, direct Codex currently has the advantage
in **setup simplicity**.

### 3. Prompt-language was better on auditability and experiment structure

Prompt-language preserved:

- persisted session state
- audit trails
- explicit gates
- intervention history
- canonical closure artifacts

That makes it materially better for **controlled software-factory experimentation**, even when the
final built slice is similar to the direct Codex result.

### 4. Early prompt-language failures were mostly runtime confounds, not factory-language failures

`A02` and `A04` should not be read as “prompt-language cannot build the product.” They are better
read as:

- `A02`: state invariant and audit-path setup failure
- `A04`: runner/runtime divergence and orphan-process cleanup failure

`A05` matters because it closes that loop: after the runtime fixes, the sequential prompt-language
lane completed successfully.

## Scientific Conclusion

The most defensible conclusion today is:

1. `prompt-language` is **not yet proven better than direct Codex at raw app-building throughput**
   for the bounded CRM core slice.
2. `prompt-language` **is already better as an auditable, restartable, governed software-factory
   control layer**.
3. The present evidence is still too small and too operationally confounded to claim global
   superiority for prompt-language.

That is a credible result, not a weak one. It tells us where the actual advantage is and what still
needs to be tested.

## What Prompt-Language Looks Good For

Based on the runs so far, prompt-language is strongest when the goal is:

- explicit completion gates
- stable artifact requirements
- postmortem-friendly evidence
- multi-agent orchestration that should survive interruptions and restarts
- experiments where closure quality matters as much as end-state code

It is not yet shown to be strictly better for:

- fastest path to one bounded code result
- lowest operational overhead
- easiest initial setup on Windows/headless environments

## Recommended Next Experiment

The next experiment should target the area where prompt-language has a plausible causal advantage.

Recommended next step:

- repeat the bounded CRM core A/B comparison on the patched runtime for `n >= 3` runs per lane
- freeze the same model, prompt budget, workspace bootstrap, and verification contract
- record per-run:
  - time to green
  - intervention count
  - verification pass/fail
  - artifact completeness
  - scorecard totals
  - failure class (`product`, `runtime`, `config`, `evidence`)

Then add one stress variant that direct Codex is weaker at:

- forced interruption / resume
- multi-agent handoff with explicit gates
- required artifact closure under the same bounded scope

## Decision

The next experiment should not ask “can prompt-language build code at all?” That is already
answered.

The next experiment should ask:

> Does prompt-language outperform direct Codex on repeatability, restartability, and governed
> completion for the same bounded factory slice after runtime fixes?
