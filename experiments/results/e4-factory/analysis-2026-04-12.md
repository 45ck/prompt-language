## Main Findings

### 1. Product capability is at least at parity on the bounded CRM core slice

In `A03`, both `prompt-language` and direct Codex completed the same bounded CRM core slice and
passed `lint`, `typecheck`, and `test`.

That means the current evidence does **not** support the claim that prompt-language blocks or
degrades core product completion for this scope.

### 2. Direct Codex is better on raw throughput in the first claim-eligible clean `S0` batch

In `A03`, the direct Codex baseline ran with less launcher/path friction and cleaner top-level
evidence files. Prompt-language needed Windows-specific launcher fixes, state-root fixes, and had
misleading run-node timeout behavior before the end-state evidence was stable.

`A07` was the directional pilot. `B02` is the first claim-eligible clean throughput batch under the
same tightened `paired-throughput-s0-external-verification` envelope.

In `B02`:

- both lanes succeeded in all four eligible pairs
- the order split was balanced at two `codex-first` pairs and two `pl-first` pairs
- direct Codex reached median `timeToGreenSec` in `719.79s`
- prompt-language reached median `timeToGreenSec` in `1514.34s`
- direct Codex was faster in both order strata:
  - `codex-first`: `673.94s` vs `1372.77s`
  - `pl-first`: `743.49s` vs `1683.37s`

Prompt-language again reached the first relevant workspace write earlier on the secondary
exploratory metric, but that did not translate into faster verified completion.

So for this bounded slice and current harness, direct Codex currently has the advantage in **setup
simplicity** and **raw time-to-green**.

### 3. Prompt-language was better on auditability and experiment structure

Prompt-language preserved:

- persisted session state
- audit trails
- explicit gates
- intervention history
- canonical closure artifacts

That makes it materially better for **controlled software-factory experimentation**, even when the
final built slice is similar to the direct Codex result.

### 4. Early prompt-language failures were mostly runtime confounds, not factory-language failures

`A02` and `A04` should not be read as “prompt-language cannot build the product.” They are better
read as:

- `A02`: state invariant and audit-path setup failure
- `A04`: runner/runtime divergence and orphan-process cleanup failure

`A05` matters because it closes that loop: after the runtime fixes, the sequential prompt-language
lane completed successfully.

## Scientific Conclusion

The most defensible conclusion today is:

1. `prompt-language` is **not better than direct Codex on raw throughput** for the bounded CRM core
   slice under the current `S0` protocol.
2. `prompt-language` **is already better as an auditable, restartable, governed software-factory
   control layer**.
3. The clean-throughput question for this specific setting now has a credible batch-level answer:
   `codex-alone-better`.
4. Any future claim that prompt-language is “better” needs to target a different dependent
   variable, such as governed recovery, restartability, or controlled multi-agent completion.

Admissibility note:

- `B02` is the current batch-level throughput evidence because it satisfies the clean-pair, order,
  and trace requirements
- `A08` through `A11` are the constituent primary-comparison runs inside that batch
- `A07` remains a useful pilot, but it is no longer the main throughput reference
- `A06` remains useful supporting timing context, but it used the older asymmetric timing envelope
- `A03` remains useful paired evidence, but it is not throughput-admissible because lane-normalized
  timing was not captured
- `A02`, `A04`, and `A05` are supporting context, not primary throughput evidence

That is a credible result, not a weak one. It tells us where the actual advantage is and what still
needs to be tested.

## What Prompt-Language Looks Good For

Based on the runs so far, prompt-language is strongest when the goal is:

- explicit completion gates
- stable artifact requirements
- postmortem-friendly evidence
- multi-agent orchestration that should survive interruptions and restarts
- experiments where closure quality matters as much as end-state code

It is not better for this `S0` setting at:

- fastest path to one bounded code result
- lowest operational overhead
- easiest initial setup on Windows/headless environments

## Recommended Next Experiment

Recommended next step:

- stop spending more cycles trying to prove raw-throughput superiority for prompt-language on this
  exact bounded clean `S0` slice unless the prompt-language control surface changes materially
- move to experiments where prompt-language has a plausible causal advantage:
- forced interruption / resume
- multi-agent handoff with explicit gates
- required artifact closure under the same bounded scope
- governed restart after mid-run failure with preserved audit state

## Decision

The next experiment should not ask “can prompt-language build code at all?” That is already
answered.

The next experiment should ask:

> Does prompt-language outperform direct Codex on restartability, governed recovery, and
> multi-agent completion quality for the same bounded factory slice after raw-throughput parity has
> already failed?
